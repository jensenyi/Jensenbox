# Token 节流协议

本协议减少重复上下文和低价值流程，不限制必要推理。选题判断、商业机制、脚本创意、视觉推演和质量裁决应保留充分推理预算。

## 1. 读取层级

### Level 0：待命

不读取文件、不巡检、不播报。只有大川提供输入或明确要求运行时才启动。

### Level 1：启动

只读取 `config.json`、`docs/core_context_pack.md` 和当前 `run_manifest.json`。

### Level 2：阶段执行

只增加读取当前阶段 Prompt、上一步 manifest/输出和本阶段直接需要的档案。

### Level 3：问题定位

仅当出现事实、平台、视觉、Seedance、GEO 或合规问题时，读取对应原始知识文件的相关章节。问题解决后把结论写入压缩规则，不在后续 Run 重复研究。

## 2. 禁止行为

- 每次启动通读 README、所有 Prompt、全部知识库和历史 Run。
- 多个线程重复研究同一热点或复制相同上下文。
- 为确认“有没有变化”反复读取未修改文件。
- 在聊天交接中粘贴完整脚本、研究包或岗位原文。
- 无输入时做主动学习、空闲巡检或状态心跳。
- 为展示过程输出大量中间说明。

## 3. 单点事实原则

- 来源事实只保存在 Research 包。
- 竞品规律只保存在 Competitor 包。
- 选题决策只保存在 selected ideas。
- 完整脚本只保存在 scripts。
- 质量问题只保存在 quality report。
- 下游通过 `content_id`、`opportunity_id` 和文件路径引用，不复制正文。

## 4. 增量工作原则

- 修改时只重写被指出的段落、镜头或字段。
- 已通过部分保持不动。
- 每轮只验证新增与变更内容；事实链受影响时才回归检查。
- 同一问题最多两轮，仍不通过则拒绝，不继续消耗。

## 5. 原始知识触发条件

- Agency 原始岗位：只有压缩职责卡无法解决具体方法问题时读取。
- Seedance Skill/reference：只有真实视频生成、运镜设计或失败排查时读取。
- 历史 Run：只有查重、复用结构或比较真实表现时读取相关一条，不批量通读。
- 联网：只有需要新事实、现行平台规则或用户明确要求最新信息时使用。

## 6. 输出纪律

- 过程更新最多说明当前阶段、发现和下一步，不复述输入。
- 中间产物保存文件，聊天只给路径和决策摘要。
- 最终汇报默认不超过：1 个核心判断、3 个成果、3 个风险/确认项。

输出可以短，但内部判断不能浅。对核心命题至少完成机制分析、反例检查和可执行性验证；存在多个合理方案时比较后再选择。

## 7. 复用机制

每次真实 Run 结束，只把稳定且可复用的新结论加入 `docs/core_context_pack.md` 或对应规则文件。一次性信息留在 Run 内，避免核心上下文持续膨胀。
